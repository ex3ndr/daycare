# Mini Apps

Mini apps are versioned static sites owned by a workspace user. Each app gets a sidebar tab in the Daycare app and is launched through a short-lived tokenized serving path.

## Overview

- Storage metadata lives in the new `mini_apps` table and follows the repo's versioned-record pattern.
- Files live under `users/<userId>/apps/<appId>/versions/<version>/`.
- The app shell fetches workspace mini apps, renders them as dynamic sidebar entries, then requests a launch path for the selected app.
- The app server serves files only from `/mini-apps/s/:token/...`, injects a `<base>` tag, and falls back to `index.html` for SPA routes.

```mermaid
flowchart LR
    A[mini_app_create / mini_app_update] --> B[MiniApps facade]
    B --> C[(mini_apps table)]
    B --> D[users/<userId>/apps/<appId>/versions/<version>/]
    E[Daycare app sidebar] --> F[GET /mini-apps]
    F --> C
    E --> G[GET /mini-apps/:id/launch]
    G --> H[short-lived mini app token]
    H --> I[/mini-apps/s/:token/...]
    I --> D
    I --> J[index.html fallback for SPA paths]
```

## Serving Model

- `GET /mini-apps/:id/launch` mints a token pinned to `{ userId, appId, version }`.
- `GET /mini-apps/s/:token/...` is public for the token lifetime and does not reuse the app session token.
- `index.html` responses inject:
  - `<base href="/mini-apps/s/:token/">` so relative asset paths stay inside the serving prefix.
  - A best-effort CSP that restricts loads to same-origin resources.
- Missing paths fall back to `index.html`, so client-side routers work without server-side route registration.

## Versioning

- Create writes version `1`.
- Update copies the previous version directory, applies file mutations, and inserts a new active metadata row.
- Delete removes the active metadata row from sidebar/API listing but keeps previous version files on disk for any already-issued short-lived launch tokens.
